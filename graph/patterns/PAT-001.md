# Behavioral Pattern: Engineers Skip Runbook Steps Based on Experience

```yaml
id: PAT-001
type: behavioral_pattern
title: "On-Call Engineers Skip Runbook Steps Based on Personal Incident History"
discovered_at: "2026-04-14T16:00:00+05:30"

pattern: |
  When MTTR pressure is high (>15 min elapsed on P1) or when the on-call
  engineer has personal experience with similar incidents, they skip runbook
  steps they've found ineffective — even if the runbook doesn't allow it.
  
  This is not reckless. It's applied institutional memory not captured anywhere.

evidence:
  - decision: DEC-001
    note: "Skipped container runtime restart based on past experience"
    runbook_deviation: true
  - decision: DEC-002
    note: "Added circuit breaker instead of increasing timeout"
    runbook_deviation: false

implications:
  - severity: high
    note: "Runbooks don't reflect operational reality"
  - severity: high
    note: "New team members won't know which steps to skip"
  - severity: critical
    note: "AI agents following runbooks literally will make worse decisions than humans"
  - severity: medium
    note: "No feedback loop from incidents to runbook updates"

recommendation: |
  1. Audit all runbooks against actual decision traces
  2. Add "skip conditions" to every runbook step
  3. Feed context graph to AI agents as primary decision input
  4. Auto-generate runbook updates from decision traces

tags: [meta-pattern, runbook-deviation, implicit-knowledge]
```
